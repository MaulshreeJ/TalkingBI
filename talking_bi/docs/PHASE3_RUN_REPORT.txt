============================================================
ARCHITECTURAL CHANGES
============================================================

1. ADAPTIVE QUERY NODE (Retry Logic)
   - Each KPI gets max 1 retry attempt
   - retry_flags dict tracks which KPIs have been retried
   - Fallback strategy: strip groupby, do plain aggregate
   - Retry results marked as "retry_success" (not "success")
   - _summary counts retries separately from first-pass successes

   Flow:
     KPI executes -> FAILS
       -> Check retry_flags[kpi_name]
       -> If not retried yet: strip time_col and segment_by
       -> Re-execute with plain aggregate
       -> Mark "retry_success" if it works
       -> Mark "failed" if retry also fails

2. DEEPPREP (Data Transformation Layer)
   - Runs inside prep_node after query results arrive
   - For each timeseries result:
     a) Converts DataFrame to records
     b) Cleans NaN/None/inf values -> replaces with 0
     c) Computes min, max, points stats
     d) Stores clean records in prepared_data
     e) Stores stats in transformed_data

3. CONDITIONAL GUARD
   - Before prep processing begins
   - If ALL KPIs failed -> abort pipeline early
   - Prevents wasted work in prep/insight/chart nodes

4. EXECUTION TRACE
   - Each node appends its name to execution_trace list
   - Final trace shows: ['query', 'prep', 'insight', 'chart']
   - Logged by executor at pipeline completion
   - Returned in API response for debugging

5. EXTRACTED _execute_kpi HELPER
   - KPI execution logic extracted to standalone function
   - Makes retry logic clean (call same function with modified params)
   - No eval(), no exec(), pure pandas operations
